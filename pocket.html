🟦 Q1 — UML & OCL

6 marks Q1a — Draw the class diagram

Include for every class: CARM
  • Classes (box with name)
  • Attributes (name : type)
  • Relationships (lines between classes)
  • Multiplicities (1, *, 0..1, 1..*)
Course Registration example: 
Student (id, name, feesOwed)
   ├── *..4 enrolledCourses
Course (code, name)
   └── *..30 students
Association: Student —*..4 — Course —*..30

6 marks Q1b — Three OCL invariants

Invariant = always true for a class (before, during, after any operation)
Syntax: context ClassName inv: condition 
-- C1: Max 4 courses per student
context Student
inv: self.enrolledCourses->size() <= 4

-- C2: Max 30 students per course
context Course
inv: self.students->size() <= 30

-- C3: Must pay fees before enrolling
context Student
inv: self.enrolledCourses->size() > 0
     implies self.feesOwed = 0

6 marks Q1c — Pre & Postconditions

Precondition (pre:) — must be TRUE before operation
Postcondition (post:) — must be TRUE after operation
@pre — refers to value BEFORE operation ran 
context Student::enrollCourse(c : Course)

pre:  self.feesOwed = 0
  and self.enrolledCourses->size() < 4
  and c.students->size() < 30

post: self.enrolledCourses =
        self.enrolledCourses@pre->including(c)
  and c.students =
        c.students@pre->including(self)

3 marks Q1d — Structural vs Behavioural

Structural constraint: about class STATE (data, attributes, associations). E.g. invariant on size of a collection.

Behavioural constraint: about OPERATIONS (what happens when called). E.g. pre/postconditions on methods.
Example pairing:
• Structural: inv: age >= 18 (always true about state)
• Behavioural: pre: balance > amount (checked when withdraw() runs)

4 marks Q1e — 2 advantages of OCL over natural language

  1. Precise and unambiguous — natural language has vague words like "many" or "soon"; OCL has defined operators (<=, size(), implies)
  2. Tool-verifiable — OCL can be checked by software tools (USE, Dresden OCL) to find contradictions; English can't be auto-checked
  3. (backup) Formal — grounded in set theory/logic, enables model-checking and code generation

🟩 Q3 — Finite State Machines

Why this is "very straightforward": Once you know the 5-tuple and how to draw a state diagram, every FSM question is mechanical. No waffle needed — just precise notation.

6 marks Q3a — The 5-tuple (formal definition)

An FSM is defined as M = (Q, Σ, δ, q₀, F)
  • Q — finite set of states (e.g. {q₀, q₁, q₂})
  • Σ (sigma) — finite set of input symbols / alphabet (e.g. {0, 1})
  • δ (delta) — transition function: Q × Σ → Q (current state + input → next state)
  • q₀ — start state (q₀ ∈ Q)
  • F — set of accepting/final states (F ⊆ Q)
Memory trick: "Queen Σends Delta Quickly From"
Q · Σ · δ · q₀ · F

6 marks Q3b — Draw a state diagram

Standard notation:
  • Circle = state (label inside: q₀, q₁, etc.)
  • Double circle = accepting/final state
  • Arrow from nowhere → starting state
  • Arrow between states labelled with input symbol
  • Every state must have a transition for every input symbol (for DFAs)
Example: DFA that accepts strings ending in "01" 
      0        0
  →(q₀)──→(q₁)──→(q₁)
    │ 1    │ 1
    ↓      ↓
  (q₀)   ((q₂))   ← double circle = accepting

States Q = {q₀, q₁, q₂}
Σ = {0, 1}
q₀ = start, F = {q₂}
δ(q₀, 0) = q₁     δ(q₀, 1) = q₀
δ(q₁, 0) = q₁     δ(q₁, 1) = q₂
δ(q₂, 0) = q₁     δ(q₂, 1) = q₀

6 marks Q3c — Trace string acceptance

To check if a string is accepted:
  1. Start at q₀
  2. Read one symbol at a time, follow the arrow
  3. After reading entire string, check if current state is in F
  4. If yes → accepted. If no → rejected.
Example — trace "101" on DFA above:
Start: q₀
Read 1 → δ(q₀,1) = q₀
Read 0 → δ(q₀,0) = q₁
Read 1 → δ(q₁,1) = q₂
End state: q₂ ∈ F → ACCEPTED ✅

4 marks Q3d — DFA vs NFA

DFANFA
TransitionsExactly one per state+inputZero, one, or many
ε transitionsNot allowedAllowed (move without input)
AcceptanceDeterministic pathAccept if any path ends in F
PowerEquivalent — every NFA can be converted to a DFA

3 marks Q3e — Applications / limitations

Uses:
  • Lexical analysis (tokenising source code)
  • Regex matching
  • Protocol design, UI state (login flows)
  • Hardware controllers (vending machines, traffic lights)
Limitations:
  • Finite memory — can't count unboundedly (e.g. can't recognise aⁿbⁿ)
  • Can't match nested structures (brackets) — need a pushdown automaton (CFG)

⚡ FSM Exam Checklist

Before handing in Q3, check:
  • ☐ State names are consistent (q₀, q₁, not Q0, Q1)
  • ☐ Start state has an arrow pointing to it
  • ☐ Accepting states have double circles
  • ☐ Every state has a transition for every symbol in Σ (DFA)
  • ☐ Δ function written as δ(q, a) = q' format
  • ☐ If asked, verify with at least one string trace

🟥 Q5 — Quality & Testing

4 marks Q5a — Reliability risks (scenario-based)

Formula: Bad input → Wrong decision → Real-world consequence

Two risks to bank:
  1. Stale/delayed data → system acts on outdated readings → wrong timing decisions → congestion/accidents
  2. Missing/corrupted data → system defaults or freezes → unsafe state (e.g. all-red stuck signals) → collision risk

4 marks Q5b — Fault tolerance strategies

  1. Redundant sensors — multiple sensors per location, voting/majority logic discards outliers (hardware redundancy)
  2. Graceful degradation — if sensors fail, system falls back to safe default cycle (fixed timing) rather than crashing
  3. (backup) Watchdog timer — monitors sensor response; triggers fallback if no data in X seconds

4 marks Q5c — Functional vs Non-Functional

FunctionalNon-Functional
TestsWHAT the system doesHOW WELL it does it
Example"Light turns red when pedestrian presses button""Light changes within 500ms of sensor input"
VerifiesRequirements are metQuality attributes (speed, reliability, security)

4 marks Q5d — V&V techniques (pick 3)

  • Code inspection / peer review — static, manual reading for defects before execution
  • Systematic testing — unit → integration → system → acceptance; black-box + white-box
  • Formal methods / model checking — mathematical proof that code meets spec (e.g. OCL, Z notation)

4 marks Q5e — Single metric limitation ⭐ (module leader specifically flagged this)

Structure your answer in 3 parts:
  1. Define "single metric": one numerical value meant to represent an entire quality attribute (1 mark)
  2. Maintainability is multi-dimensional (2 marks): readability, modularity, testability, documentation. No single number captures all. E.g. low cyclomatic complexity alone doesn't mean maintainable if variable names are awful.
  3. Reliability is multi-dimensional (2 marks): MTBF, failure rate, recovery time, availability. A system with high MTBF but slow recovery is still poor reliability. Need a composite of metrics.

3 marks Unit testing in high-integrity software

  • Early defect detection — bugs caught at unit level cost 10x less than at system level
  • Regression safety net — changes can be verified instantly, preserving past certification
  • Certification support — DO-178C, IEC 61508 require documented test coverage

🟩 Q3 — Finite State Machines

Scenario: Mock Q3 centres on language L = { w ∈ {a,b}* | no two adjacent symbols are identical }.
Q3a 5 marks — What is an FSM? Describe main elements + how they process input.
An FSM (Finite State Machine) is a mathematical model of computation with a finite number of states that processes input one symbol at a time.

Main elements (5-tuple M = (Q, Σ, δ, q₀, F)):
  • Q — finite set of states
  • Σ — alphabet of input symbols
  • δ — transition function: Q × Σ → Q
  • q₀ — start state
  • F — set of accepting (final) states
How they work together: The machine starts at q₀, reads the input string symbol by symbol, using δ to move to the next state each time. After reading the full string, if the current state is in F, the string is accepted; otherwise rejected.

Mark breakdown: 2 marks (definition) + 2 marks (lists elements) + 1 mark (processing explanation).
Q3b 4 marks — Difference between DFA and NFA.
DFA (Deterministic): for every (state, input symbol) pair, there is exactly one transition to a next state. No ε-transitions allowed. The machine's path through states is fully determined.

NFA (Non-Deterministic): a state may have zero, one, or many transitions on the same symbol. Also allows ε-transitions (move without consuming input). A string is accepted if any valid path ends in an accepting state.

Mark breakdown: 2 marks DFA definition + 2 marks NFA definition.
Q3c 6 marks — Design a DFSM for L = { w ∈ {a,b}* | no two adjacent symbols are identical }.
States: 4 total.
  • q₀ — start state (nothing read yet)
  • qₐ — last symbol was 'a'
  • q_b — last symbol was 'b'
  • q_dead — violation detected (two identical in a row)
Accepting states: F = {q₀, qₐ, q_b} (empty string is accepted, any legal trailing symbol is accepted)

Transition table: 
δ(q₀,    a) = qₐ         δ(q₀,    b) = q_b
δ(qₐ,    a) = q_dead     δ(qₐ,    b) = q_b
δ(q_b,   a) = qₐ         δ(q_b,   b) = q_dead
δ(q_dead,a) = q_dead     δ(q_dead,b) = q_dead
Diagram sketch: 
        a           b
   →(q₀)─────→(qₐ)────→(q_b)─── accepting chain
        │ b    │ a      │ b
        │      ↓ b      ↓ a
        │    (q_b)    (qₐ)
        ↓ a
       (qₐ)
    (q_dead non-accepting, traps on any input)
Mark breakdown:
  • 1 mark — Correct start state
  • 2 marks — Set of states representing last symbol read
  • 1 mark — Correct accepting state(s)
  • 2 marks — Correct transitions for a and b
Q3d 5 marks — Explain what L generates + 2 strings in L and 2 not in L.
L contains all strings over the alphabet {a, b} where no two identical symbols appear next to each other. In other words, the string alternates between 'a' and 'b' (though not necessarily starting with one or the other).

Strings IN L:
  • "ab" ✅ (a then b, no repetition)
  • "bab" ✅ (b-a-b, alternating)
  • (also acceptable: "a", "b", "", "abab", "baba")
Strings NOT in L:
  • "aa" ❌ (two a's adjacent)
  • "bba" ❌ (two b's at start)
  • (also invalid: "abba", "aab", "bbb")
Mark breakdown: 2 marks (correct explanation) + 1.5 marks (2 valid in-L) + 1.5 marks (2 valid not-in-L).
Q3e 5 marks — What information must a DFSM remember to recognise L? Why sufficient?
The DFSM only needs to remember the last symbol read. That is why the design above has one state per possible "last symbol" (qₐ, q_b, plus q₀ for empty string).

Why this is sufficient: The rule "no two adjacent symbols are identical" is a local constraint — it only compares the next symbol to the one immediately before it. There is no need to remember the full history of the string, count symbols, or track positions elsewhere. As long as the machine knows the last symbol it saw, it can correctly decide whether the next symbol breaks the rule.

This is why a DFSM (with finite memory) is enough — a more powerful model like a pushdown automaton is only required when the constraint depends on unbounded counting or nested structure (e.g. matching parentheses).

Mark breakdown:
  • 2 marks — Must remember last symbol read
  • 2 marks — Why that's enough for the adjacency check
  • 1 mark — Clear reasoning about finite-memory sufficiency

🟥 Q5 — Smart Traffic Light System

Scenario: Smart traffic light system adjusts signal timings based on road sensor data. Logic: high density → green stays longer; no vehicles → change quicker. Failures could cause accidents.
Q5a 4 marks — Two software reliability risks from inaccurate or delayed sensor data.
  1. Incorrect traffic light timing due to inaccurate data. If the sensor misreads density (e.g. reports "high" when the road is empty), the system holds a green signal for an empty lane while other lanes queue. This leads to unnecessary delays, longer journey times, and user frustration.
  2. Delayed response causing congestion or accidents. If sensor data arrives late, the system acts on stale readings — e.g. still sees the junction as clear when vehicles have just arrived. This can result in signals changing too slowly, causing gridlock, or changing at the wrong moment, creating collision risk at the junction.
Mark breakdown: 2 marks per clear risk with cause → consequence.
Q5b 4 marks — Two design strategies to make the software fault tolerant.
  1. Redundant sensors / cross-checking inputs. Deploy multiple sensors per junction (hardware redundancy) so readings can be compared. If one sensor reports a value that disagrees with the others, voting/majority logic discards the outlier. This prevents a single faulty sensor from corrupting decisions.
  2. Graceful degradation with fallback logic. If all sensors fail or data goes missing, the software reverts to a safe default cycle (e.g. fixed timing intervals, like a traditional traffic light). This ensures the junction keeps operating safely rather than freezing or issuing unsafe signals.
Mark breakdown: 2 marks per strategy with clear explanation.
Q5c 4 marks — Functional vs non-functional testing + example of each.
Functional testing checks whether the system behaves according to its requirements — i.e. what it does.
Example: Verify that when a vehicle is detected on the minor road, the traffic light correctly switches from red to green within the stated logic.

Non-functional testing checks how well the system performs — qualities like speed, reliability, scalability, usability.
Example: Measure response time — confirm the signal changes within 500ms of receiving sensor input, even under peak traffic load.

Mark breakdown: 2 marks functional (def + example) + 2 marks non-functional (def + example).
Q5d 6 marks — Three V&V techniques to ensure safe operation.
  1. Simulation / scenario-based testing. Run the software against realistic recorded or synthetic traffic scenarios (rush hour, emergency vehicle, sensor failure). Exposes edge cases before deployment and verifies the system handles unusual inputs safely.
  2. Formal verification / model checking. Mathematically prove that the logic rules satisfy required safety properties (e.g. "no two conflicting signals are ever green simultaneously"). Tools like model checkers exhaustively explore the state space, catching issues testing might miss.
  3. Code inspections / safety audits. Peer review of the source code and design artefacts by qualified engineers, looking for hazards, coding errors, or deviations from safety standards. Particularly important for high-integrity systems where certification is required.
Mark breakdown: 2 marks per technique × 3 = 6 marks.
Q5e 5 marks — Limitations of using a single metric for quality attributes like maintainability or reliability. ⭐
A single metric is one numerical measurement intended to represent an entire quality attribute. For a complex attribute, a single number cannot capture everything.

Maintainability example: A single metric such as cyclomatic complexity measures control-flow branching, but ignores other dimensions of maintainability — code readability (variable names, comments), modularity (separation of concerns), and testability (how easily behaviour can be verified). Two systems with identical complexity scores can have vastly different real maintainability.

Reliability example: A metric such as MTBF (mean time between failures) doesn't capture recovery time, severity of failure, or availability. A system with excellent MTBF but catastrophically slow recovery is still unreliable in practice.

Conclusion: Complex quality attributes should be evaluated using multiple metrics combined with qualitative assessment (code reviews, user feedback, architecture analysis) to form a complete picture.

Mark breakdown:
  • 2 marks — Single metric can't capture all aspects
  • 2 marks — Concrete example (maintainability or reliability)
  • 1 mark — Notes multiple metrics + qualitative evaluation needed
Q5f 2 marks — Purpose of unit testing in high-integrity software.
Unit testing verifies that individual software components (units) — functions, classes, modules — work correctly in isolation. In high-integrity software it has two critical roles:
  1. Ensures each component behaves as specified before integration.
  2. Helps detect and isolate faults early, supporting regression safety nets and certification evidence (e.g. DO-178C, IEC 61508).
Mark breakdown: 1 mark (components work correctly) + 1 mark (early fault detection / reliability).

⚡ Rapid-Fire Recall

OCL:
  • context X inv: Y — binds invariant to class
  • implies — logical "if A then B"
  • ->size() — collection size
  • ->including(x) — add to collection
  • @pre — value before operation
  • self — the current object
FSM keywords:
  • Q — set of states
  • Σ — alphabet (input symbols)
  • δ — transition function (state × symbol → state)
  • q₀ — start state
  • F — accepting (final) states
  • DFA — one transition per input (deterministic)
  • NFA — many transitions allowed (non-deterministic) + ε moves
  • Double circle = accepting state in diagram
  • Limitation — finite memory, can't count unboundedly
Testing keywords:
  • Functional = WHAT
  • Non-functional = HOW WELL
  • Verification = "building it right" (code matches spec)
  • Validation = "building the right thing" (spec matches need)
  • MTBF = mean time between failures
  • Fault tolerance = continues operating under failure
  • Graceful degradation = reduced function, still safe
Common mistakes to avoid:
  • Writing invariants without context and inv:
  • Missing transitions — every DFA state needs a transition for every symbol
  • Forgetting the double circle for accepting states
  • Writing δ function without showing (state, input) → next state format
  • Answering Q5a with "affects reliability" — too vague, say WHAT goes wrong
  • Forgetting to define the term before applying it
If you only remember 10 things:
  1. Invariant = always true, pre = before, post = after
  2. OCL syntax: context X inv:
  3. implies = logical "if...then"
  4. FSM = (Q, Σ, δ, q₀, F) — memorise this tuple cold
  5. Double circle = accepting state, arrow-in = start state
  6. DFA = 1 transition per input; NFA = many + ε allowed
  7. Trace strings: start at q₀, follow arrows, check if end ∈ F
  8. Reliability risk: stale data → wrong decision → consequence
  9. Fault tolerance: redundancy + graceful degradation
  10. Single metric: DEFINE first, then multi-dimensional argument
