6 marks Q1c — Pre & Postconditions

Precondition (pre:) — must be TRUE before operation
Postcondition (post:) — must be TRUE after operation
@pre — refers to value BEFORE operation ran 
context Student::enrollCourse(c : Course)

pre:  self.feesOwed = 0
  and self.enrolledCourses->size() < 4

3 marks Q1d — Structural vs Behavioural

Structural constraint: about class STATE (data, attributes, associations). E.g. invariant on size of a collection.

Behavioural constraint: about OPERATIONS (what happens when called). E.g. pre/postconditions on methods.
Example pairing:
• Structural: inv: age >= 18 (always true about state)
• Behavioural: pre: balance > amount (checked when withdraw() runs)

4 marks Q1e — 2 advantages of OCL over natural language

  1. Precise and unambiguous — natural language has vague words like "many" or "soon"; OCL has defined operators (<=, size(), implies)
  2. Tool-verifiable — OCL can be checked by software tools (USE, Dresden OCL) to find contradictions; English can't be auto-checked
  3. (backup) Formal — grounded in set theory/logic, enables model-checking and code generation

🟩 Q2 — Concurrency

Why this plays to your strengths: Q2 is 100% terminology + explanation. No diagrams, no formal notation — just name the technique and explain it in plain English. Every mark is available through clear prose.

5 marks Q2a — Why shared resources need synchronization

When multiple threads access a shared resource concurrently, and at least one thread writes to it, the outcome becomes unpredictable. Without synchronization, interleaved operations can leave data in an inconsistent or corrupted state.
  • Shared memory (counters, queues, caches)
  • Shared databases / files
  • Hardware resources (printers, sensors)
Food delivery example: two threads both try to assign the same driver to different orders at the same millisecond. Without locking the driver database, both reads see the driver as available, both writes mark them assigned — double-booking, lost money.

5 marks Q2b — Race condition

Race condition: two or more threads access a shared resource at the same time, and the final result depends on the unpredictable order of their operations → incorrect or corrupted data.
In the order queue: two threads simultaneously dequeue the same order entry because neither locked the queue first. Both attempt to process it — one delivery is lost, or the same order is dispatched twice.

Consequence: lost updates, duplicated work, inconsistent state across the system.

6 marks Q2c — Two synchronization mechanisms

MechanismHow it worksMaintains consistency by...
Mutex (Mutual Exclusion Lock) Binary lock — only one thread can hold it at a time. Other threads block until it's released. Ensuring only one thread modifies the order queue at any moment — no interleaved writes.
Semaphore Counter-based — allows up to N threads through. Each acquire decrements, each release increments. Limiting concurrent access to a resource pool (e.g. 5 DB connections) — prevents overload while allowing parallelism.
Monitor High-level construct — methods are auto-locked on entry and unlocked on exit (Java synchronized). Encapsulating lock management inside the object — programmers can't forget to release the lock.

5 marks Q2d — Thread pools

Thread pool: a fixed-size pool of pre-created worker threads that are reused across tasks. When a new task arrives, it's handed to an idle worker; if all workers are busy, it queues until one is free. Threads are not destroyed after each task.
Two advantages for high-traffic systems:
  1. Reduced thread creation overhead — creating a new thread is expensive (memory allocation, OS registration). Reusing existing threads avoids this cost for every task.
  2. Better resource management — the pool size caps concurrency. This prevents the system from spawning thousands of threads during peak traffic, which would exhaust memory and collapse performance.

4 marks Q2e — Why excessive threading hurts performance

  1. Context-switching overhead — the CPU spends increasing time saving/restoring thread state rather than executing real work. Past a threshold, adding more threads reduces throughput.
  2. Memory exhaustion + lock contention — every thread has its own stack (≈1 MB). Thousands of threads exhaust RAM. Plus, more threads competing for the same locks means most of them sit idle waiting — no real parallelism gained.

Bonus: Deadlock (may appear per Dr Ramzan)

Deadlock: two or more threads each hold a lock the other needs, so both wait forever.

Coffman's 4 conditions (all must hold): mutual exclusion · hold-and-wait · no preemption · circular wait

Prevention: enforce lock ordering — always acquire locks in a globally agreed sequence, so no cycle can form.

🟥 Q5 — Quality & Testing

4 marks Q5a — Reliability risks (scenario-based)

Formula: Bad input → Wrong decision → Real-world consequence

Two risks to bank:
  1. Stale/delayed data → system acts on outdated readings → wrong timing decisions → congestion/accidents
  2. Missing/corrupted data → system defaults or freezes → unsafe state (e.g. all-red stuck signals) → collision risk

4 marks Q5b — Fault tolerance strategies

  1. Redundant sensors — multiple sensors per location, voting/majority logic discards outliers (hardware redundancy)
  2. Graceful degradation — if sensors fail, system falls back to safe default cycle (fixed timing) rather than crashing
  3. (backup) Watchdog timer — monitors sensor response; triggers fallback if no data in X seconds

  • Systematic testing — unit → integration → system → acceptance; black-box + white-box
  • Formal methods / model checking — mathematical proof that code meets spec (e.g. OCL, Z notation)

4 marks Q5e — Single metric limitation ⭐ (module leader specifically flagged this)

Structure your answer in 3 parts:
  1. Define "single metric": one numerical value meant to represent an entire quality attribute (1 mark)
  2. Maintainability is multi-dimensional (2 marks): readability, modularity, testability, documentation. No single number captures all. E.g. low cyclomatic complexity alone doesn't mean maintainable if variable names are awful.
  3. Reliability is multi-dimensional (2 marks): MTBF, failure rate, recovery time, availability. A system with high MTBF but slow recovery is still poor reliability. Need a composite of metrics.

3 marks Unit testing in high-integrity software

  • Early defect detection — bugs caught at unit level cost 10x less than at system level
  • Regression safety net — changes can be verified instantly, preserving past certification

🟩 Q2 — Food Delivery Platform

Scenario: A food delivery platform processes thousands of orders simultaneously. Each thread verifies payment, assigns a driver, and updates the restaurant queue. During peak hours, threads contend for the driver database and order queue.
Q2a 5 marks — Why shared resources need synchronization (illustrate with driver database or order queue).
In multithreaded systems, multiple threads execute concurrently and may read from or write to the same shared resource at the same time. Without synchronization, these operations interleave unpredictably, leading to inconsistent state or data corruption.

Example — driver database: Two threads process two different orders simultaneously. Both read that driver #42 is "available", both update the record to assign them. Without synchronization, both succeed → the driver is double-booked, one delivery will fail.

Synchronization mechanisms (locks, mutexes, semaphores) enforce exclusive or controlled access so that operations on the shared resource happen in a consistent order, preserving data integrity.

Mark breakdown: 2 marks (concurrent access causes corruption) + 2 marks (driver/queue example) + 1 mark (names sync mechanism).
Q2b 5 marks — Define a race condition and describe how it could occur in the order queue.
A race condition occurs when two or more threads access a shared resource concurrently, and the final outcome depends on the unpredictable order in which their operations interleave. This leads to incorrect or inconsistent behaviour that may vary between runs.

In the order queue: suppose Thread A and Thread B both attempt to dequeue the next order. Both read the head-of-queue pointer, both try to remove the same order, and both mark it "in progress". One order is processed twice (duplicate delivery), while the next order is skipped entirely because the pointer was only advanced once.

Consequence: lost updates, duplicated work, inconsistent queue state, corrupted system records — all of which undermine customer trust and operational reliability.

Mark breakdown: 2 marks (clear definition) + 2 marks (queue example with interleaving) + 1 mark (consequence).
Q2c 6 marks — Two synchronization mechanisms + how each maintains data consistency.
Mechanism 1 — Mutex (Mutual Exclusion Lock):
A mutex is a binary lock that allows only one thread at a time to enter a critical section. A thread must acquire the lock before reading/writing the shared resource; any other thread attempting to acquire the same lock is blocked until the lock is released.
Maintains consistency by: preventing interleaved writes to the order queue — only one thread can modify the queue at any moment, so race conditions are eliminated.

Mechanism 2 — Semaphore:
A semaphore is a counter-based synchronization primitive. It allows up to N concurrent threads (acquire decrements the counter, release increments it). If the counter reaches zero, further threads block until one releases.
Maintains consistency by: controlling access to a pool of identical resources (e.g. 10 database connections) — multiple threads can work in parallel, but not so many that the resource is overwhelmed.

(Monitor — backup answer): a high-level construct where methods are automatically locked on entry and unlocked on exit (e.g. Java synchronized). Encapsulates lock management inside the object so programmers can't forget to release.

Mark breakdown: 3 marks per mechanism × 2 = 6 marks.
Q2d 5 marks — How a thread pool works + two advantages for high-traffic systems.
A thread pool is a pre-created, fixed-size collection of worker threads that are reused across many tasks. When a new task arrives (e.g. a new order), it is submitted to the pool's queue; an idle worker picks it up, executes it, and returns to the pool ready for the next task. Threads are not destroyed between tasks.

Advantage 1 — Reduced thread creation overhead: Creating a new thread is expensive (stack allocation, OS registration, initialisation). In a high-traffic system with thousands of orders per second, per-task thread creation would dominate CPU time. Reusing existing threads eliminates this cost.

Advantage 2 — Better resource management / bounded concurrency: The pool caps the maximum number of simultaneous threads. This prevents the system from spawning an unbounded number of threads during a traffic spike, which would otherwise exhaust memory, cause lock contention, and crash the service.

Mark breakdown: 2 marks (how it works) + 2 marks (advantage 1) + 1 mark (advantage 2).
Q2e 4 marks — Two reasons excessive threading hurts system performance.
Reason 1 — Context-switching overhead:
Each time the operating system switches between threads, it must save the current thread's state (registers, program counter, stack pointer) and load the next thread's state. With too many active threads, the CPU spends an increasing fraction of its time context-switching rather than executing real work — throughput falls as thread count grows past the sweet spot.

Reason 2 — Increased memory use and lock contention:
Every thread has its own stack (typically ≈1 MB). Thousands of threads consume gigabytes of RAM purely for overhead. Additionally, more threads competing for the same locks means most of them sit waiting for locks to be released — no real parallelism is gained, and the "critical section" becomes a bottleneck.

Mark breakdown: 2 marks per reason × 2 = 4 marks.

🟥 Q5 — Smart Traffic Light System

Scenario: Smart traffic light system adjusts signal timings based on road sensor data. Logic: high density → green stays longer; no vehicles → change quicker. Failures could cause accidents.
Q5a 4 marks — Two software reliability risks from inaccurate or delayed sensor data.
  1. Incorrect traffic light timing due to inaccurate data. If the sensor misreads density (e.g. reports "high" when the road is empty), the system holds a green signal for an empty lane while other lanes queue. This leads to unnecessary delays, longer journey times, and user frustration.
  2. Delayed response causing congestion or accidents. If sensor data arrives late, the system acts on stale readings — e.g. still sees the junction as clear when vehicles have just arrived. This can result in signals changing too slowly, causing gridlock, or changing at the wrong moment, creating collision risk at the junction.
Mark breakdown: 2 marks per clear risk with cause → consequence.
Q5b 4 marks — Two design strategies to make the software fault tolerant.
  1. Redundant sensors / cross-checking inputs. Deploy multiple sensors per junction (hardware redundancy) so readings can be compared. If one sensor reports a value that disagrees with the others, voting/majority logic discards the outlier. This prevents a single faulty sensor from corrupting decisions.
  2. Graceful degradation with fallback logic. If all sensors fail or data goes missing, the software reverts to a safe default cycle (e.g. fixed timing intervals, like a traditional traffic light). This ensures the junction keeps operating safely rather than freezing or issuing unsafe signals.
Mark breakdown: 2 marks per strategy with clear explanation.
Q5c 4 marks — Functional vs non-functional testing + example of each.
Functional testing checks whether the system behaves according to its requirements — i.e. what it does.
Example: Verify that when a vehicle is detected on the minor road, the traffic light correctly switches from red to green within the stated logic.

Non-functional testing checks how well the system performs — qualities like speed, reliability, scalability, usability.
Example: Measure response time — confirm the signal changes within 500ms of receiving sensor input, even under peak traffic load.

Mark breakdown: 2 marks functional (def + example) + 2 marks non-functional (def + example).
Q5d 6 marks — Three V&V techniques to ensure safe operation.
  1. Simulation / scenario-based testing. Run the software against realistic recorded or synthetic traffic scenarios (rush hour, emergency vehicle, sensor failure). Exposes edge cases before deployment and verifies the system handles unusual inputs safely.
  2. Formal verification / model checking. Mathematically prove that the logic rules satisfy required safety properties (e.g. "no two conflicting signals are ever green simultaneously"). Tools like model checkers exhaustively explore the state space, catching issues testing might miss.
  3. Code inspections / safety audits. Peer review of the source code and design artefacts by qualified engineers, looking for hazards, coding errors, or deviations from safety standards. Particularly important for high-integrity systems where certification is required.
Mark breakdown: 2 marks per technique × 3 = 6 marks.
Q5e 5 marks — Limitations of using a single metric for quality attributes like maintainability or reliability. ⭐
A single metric is one numerical measurement intended to represent an entire quality attribute. For a complex attribute, a single number cannot capture everything.

Maintainability example: A single metric such as cyclomatic complexity measures control-flow branching, but ignores other dimensions of maintainability — code readability (variable names, comments), modularity (separation of concerns), and testability (how easily behaviour can be verified). Two systems with identical complexity scores can have vastly different real maintainability.

Reliability example: A metric such as MTBF (mean time between failures) doesn't capture recovery time, severity of failure, or availability. A system with excellent MTBF but catastrophically slow recovery is still unreliable in practice.

Conclusion: Complex quality attributes should be evaluated using multiple metrics combined with qualitative assessment (code reviews, user feedback, architecture analysis) to form a complete picture.

Mark breakdown:
  • 2 marks — Single metric can't capture all aspects
  • 2 marks — Concrete example (maintainability or reliability)
  • 1 mark — Notes multiple metrics + qualitative evaluation needed
Q5f 2 marks — Purpose of unit testing in high-integrity software.
Unit testing verifies that individual software components (units) — functions, classes, modules — work correctly in isolation. In high-integrity software it has two critical roles:
  1. Ensures each component behaves as specified before integration.
  2. Helps detect and isolate faults early, supporting regression safety nets and certification evidence (e.g. DO-178C, IEC 61508).
Mark breakdown: 1 mark (components work correctly) + 1 mark (early fault detection / reliability).

🆘 Q3 FSM — Emergency Backup

Use only if: Q1 or Q2 has a weirdly phrased scenario you can't parse, or you run out of ideas. Otherwise stick with Q1 + Q2 + Q5 — they play to your strengths.
The FSM 5-tuple (cold memorisation)
M = (Q, Σ, δ, q₀, F)
  • Q — finite set of states
  • Σ (sigma) — alphabet (input symbols)
  • δ (delta) — transition function Q × Σ → Q
  • q₀ — start state
  • F — set of accepting (final) states
DFA vs NFA (quick diff)
DFA: exactly one transition per (state, symbol). No ε-transitions. Deterministic path.

NFA: zero, one, or many transitions per (state, symbol). Allows ε-transitions (move without input). Accept if any path ends in F.

Both are equivalent in power — every NFA can be converted to a DFA.
How to draw a DFSM (diagram rules)
  • Circle = state (labelled q₀, q₁…)
  • Double circle = accepting state
  • Arrow from nowhere → start state
  • Arrow between states labelled with input symbol
  • Every state must have a transition for every symbol in Σ (DFA)
Example — DFSM that accepts "no two adjacent identical symbols" over {a,b}: Transition function: 
δ(q₀, a)    = qₐ       δ(q₀, b)    = q_b
δ(qₐ, a)    = q_dead   δ(qₐ, b)    = q_b
δ(q_b, a)   = qₐ       δ(q_b, b)   = q_dead
δ(q_dead,·) = q_dead   (traps)
How to trace a string
1. Start at q₀
2. For each symbol, apply δ to move to next state
3. After consuming whole string, check if current state ∈ F
4. If yes → accepted; if no → rejected

Always show your trace step-by-step in the answer — it gets partial marks even if the DFSM design is slightly off.
If you need to attempt Q3 cold: answer the terminology sub-parts first (Q3a, Q3b, Q3d, Q3e — all prose) and skip the DFSM diagram (Q3c) if you can't visualise it. You can still scrape ~12–15 marks from Q3 without drawing the diagram perfectly.
