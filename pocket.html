🟦 Q1 — UML & OCL

6 marks Q1a — Draw the class diagram

Include for every class: CARM
  • Classes (box with name)
  • Attributes (name : type)
  • Relationships (lines between classes)
  • Multiplicities (1, *, 0..1, 1..*)
Course Registration example: 
Student (id, name, feesOwed)
   ├── *..4 enrolledCourses
Course (code, name)
   └── *..30 students
Association: Student —*..4 — Course —*..30

6 marks Q1b — Three OCL invariants

Invariant = always true for a class (before, during, after any operation)
Syntax: context ClassName inv: condition 
-- C1: Max 4 courses per student
context Student
inv: self.enrolledCourses->size() <= 4

-- C2: Max 30 students per course
context Course
inv: self.students->size() <= 30

-- C3: Must pay fees before enrolling
context Student
inv: self.enrolledCourses->size() > 0
     implies self.feesOwed = 0

6 marks Q1c — Pre & Postconditions

Precondition (pre:) — must be TRUE before operation
Postcondition (post:) — must be TRUE after operation
@pre — refers to value BEFORE operation ran 
context Student::enrollCourse(c : Course)

pre:  self.feesOwed = 0
  and self.enrolledCourses->size() < 4
  and c.students->size() < 30

post: self.enrolledCourses =
        self.enrolledCourses@pre->including(c)
  and c.students =
        c.students@pre->including(self)

3 marks Q1d — Structural vs Behavioural

Structural constraint: about class STATE (data, attributes, associations). E.g. invariant on size of a collection.

Behavioural constraint: about OPERATIONS (what happens when called). E.g. pre/postconditions on methods.
Example pairing:
• Structural: inv: age >= 18 (always true about state)
• Behavioural: pre: balance > amount (checked when withdraw() runs)

4 marks Q1e — 2 advantages of OCL over natural language

  1. Precise and unambiguous — natural language has vague words like "many" or "soon"; OCL has defined operators (<=, size(), implies)
  2. Tool-verifiable — OCL can be checked by software tools (USE, Dresden OCL) to find contradictions; English can't be auto-checked
  3. (backup) Formal — grounded in set theory/logic, enables model-checking and code generation

🟩 Q3 — Finite State Machines

Why this is "very straightforward": Once you know the 5-tuple and how to draw a state diagram, every FSM question is mechanical. No waffle needed — just precise notation.

6 marks Q3a — The 5-tuple (formal definition)

An FSM is defined as M = (Q, Σ, δ, q₀, F)
  • Q — finite set of states (e.g. {q₀, q₁, q₂})
  • Σ (sigma) — finite set of input symbols / alphabet (e.g. {0, 1})
  • δ (delta) — transition function: Q × Σ → Q (current state + input → next state)
  • q₀ — start state (q₀ ∈ Q)
  • F — set of accepting/final states (F ⊆ Q)
Memory trick: "Queen Σends Delta Quickly From"
Q · Σ · δ · q₀ · F

6 marks Q3b — Draw a state diagram

Standard notation:
  • Circle = state (label inside: q₀, q₁, etc.)
  • Double circle = accepting/final state
  • Arrow from nowhere → starting state
  • Arrow between states labelled with input symbol
  • Every state must have a transition for every input symbol (for DFAs)
Example: DFA that accepts strings ending in "01" 
      0        0
  →(q₀)──→(q₁)──→(q₁)
    │ 1    │ 1
    ↓      ↓
  (q₀)   ((q₂))   ← double circle = accepting

States Q = {q₀, q₁, q₂}
Σ = {0, 1}
q₀ = start, F = {q₂}
δ(q₀, 0) = q₁     δ(q₀, 1) = q₀
δ(q₁, 0) = q₁     δ(q₁, 1) = q₂
δ(q₂, 0) = q₁     δ(q₂, 1) = q₀

6 marks Q3c — Trace string acceptance

To check if a string is accepted:
  1. Start at q₀
  2. Read one symbol at a time, follow the arrow
  3. After reading entire string, check if current state is in F
  4. If yes → accepted. If no → rejected.
Example — trace "101" on DFA above:
Start: q₀
Read 1 → δ(q₀,1) = q₀
Read 0 → δ(q₀,0) = q₁
Read 1 → δ(q₁,1) = q₂
End state: q₂ ∈ F → ACCEPTED ✅

4 marks Q3d — DFA vs NFA

DFANFA
TransitionsExactly one per state+inputZero, one, or many
ε transitionsNot allowedAllowed (move without input)
AcceptanceDeterministic pathAccept if any path ends in F
PowerEquivalent — every NFA can be converted to a DFA

3 marks Q3e — Applications / limitations

Uses:
  • Lexical analysis (tokenising source code)
  • Regex matching
  • Protocol design, UI state (login flows)
  • Hardware controllers (vending machines, traffic lights)
Limitations:
  • Finite memory — can't count unboundedly (e.g. can't recognise aⁿbⁿ)
  • Can't match nested structures (brackets) — need a pushdown automaton (CFG)

⚡ FSM Exam Checklist

Before handing in Q3, check:
  • ☐ State names are consistent (q₀, q₁, not Q0, Q1)
  • ☐ Start state has an arrow pointing to it
  • ☐ Accepting states have double circles
  • ☐ Every state has a transition for every symbol in Σ (DFA)
  • ☐ Δ function written as δ(q, a) = q' format
  • ☐ If asked, verify with at least one string trace

🟥 Q5 — Quality & Testing

4 marks Q5a — Reliability risks (scenario-based)

Formula: Bad input → Wrong decision → Real-world consequence

Two risks to bank:
  1. Stale/delayed data → system acts on outdated readings → wrong timing decisions → congestion/accidents
  2. Missing/corrupted data → system defaults or freezes → unsafe state (e.g. all-red stuck signals) → collision risk

4 marks Q5b — Fault tolerance strategies

  1. Redundant sensors — multiple sensors per location, voting/majority logic discards outliers (hardware redundancy)
  2. Graceful degradation — if sensors fail, system falls back to safe default cycle (fixed timing) rather than crashing
  3. (backup) Watchdog timer — monitors sensor response; triggers fallback if no data in X seconds

4 marks Q5c — Functional vs Non-Functional

FunctionalNon-Functional
TestsWHAT the system doesHOW WELL it does it
Example"Light turns red when pedestrian presses button""Light changes within 500ms of sensor input"
VerifiesRequirements are metQuality attributes (speed, reliability, security)

4 marks Q5d — V&V techniques (pick 3)

  • Code inspection / peer review — static, manual reading for defects before execution
  • Systematic testing — unit → integration → system → acceptance; black-box + white-box
  • Formal methods / model checking — mathematical proof that code meets spec (e.g. OCL, Z notation)

4 marks Q5e — Single metric limitation ⭐ (module leader specifically flagged this)

Structure your answer in 3 parts:
  1. Define "single metric": one numerical value meant to represent an entire quality attribute (1 mark)
  2. Maintainability is multi-dimensional (2 marks): readability, modularity, testability, documentation. No single number captures all. E.g. low cyclomatic complexity alone doesn't mean maintainable if variable names are awful.
  3. Reliability is multi-dimensional (2 marks): MTBF, failure rate, recovery time, availability. A system with high MTBF but slow recovery is still poor reliability. Need a composite of metrics.

3 marks Unit testing in high-integrity software

  • Early defect detection — bugs caught at unit level cost 10x less than at system level
  • Regression safety net — changes can be verified instantly, preserving past certification
  • Certification support — DO-178C, IEC 61508 require documented test coverage

⚡ Rapid-Fire Recall

OCL:
  • context X inv: Y — binds invariant to class
  • implies — logical "if A then B"
  • ->size() — collection size
  • ->including(x) — add to collection
  • @pre — value before operation
  • self — the current object
FSM keywords:
  • Q — set of states
  • Σ — alphabet (input symbols)
  • δ — transition function (state × symbol → state)
  • q₀ — start state
  • F — accepting (final) states
  • DFA — one transition per input (deterministic)
  • NFA — many transitions allowed (non-deterministic) + ε moves
  • Double circle = accepting state in diagram
  • Limitation — finite memory, can't count unboundedly
Testing keywords:
  • Functional = WHAT
  • Non-functional = HOW WELL
  • Verification = "building it right" (code matches spec)
  • Validation = "building the right thing" (spec matches need)
  • MTBF = mean time between failures
  • Fault tolerance = continues operating under failure
  • Graceful degradation = reduced function, still safe
Common mistakes to avoid:
  • Writing invariants without context and inv:
  • Missing transitions — every DFA state needs a transition for every symbol
  • Forgetting the double circle for accepting states
  • Writing δ function without showing (state, input) → next state format
  • Answering Q5a with "affects reliability" — too vague, say WHAT goes wrong
  • Forgetting to define the term before applying it
If you only remember 10 things:
  1. Invariant = always true, pre = before, post = after
  2. OCL syntax: context X inv:
  3. implies = logical "if...then"
  4. FSM = (Q, Σ, δ, q₀, F) — memorise this tuple cold
  5. Double circle = accepting state, arrow-in = start state
  6. DFA = 1 transition per input; NFA = many + ε allowed
  7. Trace strings: start at q₀, follow arrows, check if end ∈ F
  8. Reliability risk: stale data → wrong decision → consequence
  9. Fault tolerance: redundancy + graceful degradation
  10. Single metric: DEFINE first, then multi-dimensional argument
