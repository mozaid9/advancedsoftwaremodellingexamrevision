Starter Glossary

Every technical term you will meet in this module, defined in plain English before it appears in the topic sections. Learn these first — the exam expects you to use them precisely.

Formal Modelling

UML (Unified Modelling Language)
A standardised set of diagram types used to describe software systems visually. Not a programming language.
Class
A blueprint for objects. Stores attributes (data) and has relationships with other classes.
Attribute
A piece of data belonging to a class (e.g., studentID, name, fees).
Multiplicity
A number on an association line showing how many objects can participate. "0..4" means zero to four.
Association
A line between two classes showing that they are connected in the system.
OCL (Object Constraint Language)
A formal text language for writing precise rules about a UML model. Not executable code.
Invariant
An OCL rule that must be true at all times (e.g., a student always has ≤ 4 courses).
Precondition
A condition that must be true before an operation is allowed to run.
Postcondition
A condition that must be true after an operation has completed.
@pre
OCL notation for "the value this had before the operation ran". Used only in postconditions.
Structural constraint
A rule about the valid state of the model (invariant). E.g., max 4 courses per student.
Behavioural constraint
A rule about how operations must behave (pre/post conditions). E.g., fees must be zero before enrolment.

Concurrency

Thread
A lightweight unit of execution within a process. Multiple threads share the same process memory.
Concurrency
Multiple tasks making progress during the same time period (not necessarily at the exact same instant).
Shared resource
Data or a device that multiple threads can access (e.g., the driver database, order queue).
Race condition
A bug where the outcome depends on the exact timing of two threads accessing shared data simultaneously.
Critical section
A code block that accesses shared data and must not be executed by two threads at the same time.
Mutex (Mutual Exclusion Lock)
A lock that only one thread can hold at a time. Other threads wait until the lock is released.
Semaphore
A counter controlling how many threads can access a resource simultaneously. A binary semaphore (0 or 1) acts like a mutex.
Monitor
A higher-level synchronisation construct that bundles a mutex and condition variables into a single object.
Thread pool
A fixed set of pre-created threads that pick up tasks from a queue rather than being created/destroyed per task.
Context switching
The OS saving one thread's state and restoring another's. Expensive when done too frequently.
Deadlock
A situation where two or more threads each hold a resource the other needs, so all are permanently blocked.

Formal Languages & Automata

Finite State Machine (FSM)
A model of computation with a finite number of states, transitions, and rules for accepting or rejecting input strings.
DFA (Deterministic Finite Automaton)
An FSM where each state has exactly one transition for each input symbol. No ambiguity.
NFA (Non-deterministic Finite Automaton)
An FSM where a state may have zero, one, or many transitions for a symbol, and may use ε-transitions (free moves).
Alphabet (Σ)
The set of valid input symbols for a language (e.g., {a, b} or {0, 1}).
Accepting / Final state
A state in which the machine accepts the input string if it has consumed all input symbols.
Transition function (δ)
The rule that says: given current state and current input symbol, which state to move to next.
Context-free grammar (CFG)
A set of rewriting rules (productions) that generate strings of a formal language. More powerful than regular expressions.
Production rule
A rule in a grammar of the form A → α, meaning non-terminal A can be replaced by string α.
Leftmost derivation
Expanding always the leftmost non-terminal first, step by step, until only terminals remain.
Parse tree
A tree showing how a grammar derives a string. The root is the start symbol; leaves are terminal symbols.

Compilers & Quality

Lexical analysis
The first phase of compilation: scanning characters and grouping them into tokens.
Token
A meaningful unit of source code: a keyword, identifier, literal, operator, or delimiter.
Symbol table
A data structure built during compilation that stores identifiers and their attributes (type, scope, address).
Bottom-up parser
A parser that reduces the input stream back to the start symbol by reversing grammar productions.
Functional testing
Testing that checks what a system does — whether outputs match requirements for given inputs.
Non-functional testing
Testing that checks how well the system performs — speed, reliability, security, usability.
Verification
"Are we building the product right?" — checking the system against its specification.
Validation
"Are we building the right product?" — checking the system meets actual user needs.
Fault tolerance
A system's ability to continue operating correctly even when some component fails.
GQM (Goal–Question–Metric)
A framework: define a goal, derive questions that measure it, then select metrics that answer the questions.
SOLID Principles
Five object-oriented design principles: Single Responsibility, Open/Closed, Liskov, Interface Segregation, Dependency Inversion.

Part c — OCL Pre/Postconditions (6 marks) · Time guide: 6 minutes

Write OCL preconditions and postconditions for an operation enrollCourse(student: Student, course: Course) to ensure that C1, C2, and C3 are enforced.

Mark scheme 2 marks – Preconditions correctly ensure C1, C2, and C3 · 2 marks – Postconditions correctly update enrolment · 2 marks – Proper syntax and clarity
Show full model answer — Q1c
Model Answer — Q1c (6 marks) 
context RegistrationSystem::enrollCourse(student: Student, course: Course)

-- Preconditions: conditions that must hold BEFORE the operation runs
pre StudentBelowLimit:
    student.enrolments->size() < 4          -- strictly less than 4, not ≤ 4

pre CourseHasSpace:
    course.enrolments->size() < 30         -- strictly less than 30

pre FeesCleared:
    student.outstandingFees = 0

-- Postconditions: conditions that must hold AFTER the operation completes
post StudentEnrolled:
    student.enrolments = student.enrolments@pre->including(course)

post CourseUpdated:
    course.enrolments->size() = course.enrolments@pre->size() + 1
Critical distinction — < vs <= The precondition uses < 4 (strictly less than 4), not <= 4. If you use <= 4 as the precondition for adding a course, a student with exactly 4 courses could still enrol, giving them 5. This loses a mark.

@pre explained: student.enrolments@pre means "the value of student.enrolments before this operation ran". The postcondition says the new set equals the old set with the new course added in.

Part d — Structural vs Behavioural Constraints (3 marks) · Time guide: 3 minutes

Describe the difference between structural constraints and behavioural constraints in OCL. Provide a brief example of each in the context of the Online Course Registration System.

Show full model answer — Q1d
Model Answer — Q1d (3 marks)

Structural constraint — a condition on the valid state of the model. It defines what must be true about the model's data at any point in time. It is expressed as an invariant.

Example: context Student inv: self.enrolments->size() <= 4 — this says that at any moment, a student's enrolment count cannot exceed 4. It constrains the model's data, not any specific operation.

Behavioural constraint — a condition on how an operation must behave. It restricts what may be true before the operation runs (precondition) or what must be true after it runs (postcondition).

Example: context RegistrationSystem::enrollCourse(...) pre: student.outstandingFees = 0 — this constrains the operation's behaviour by requiring the student's fees to be cleared before enrolment is allowed.

Part e — OCL Advantages (4 marks) · Time guide: 4 minutes

Discuss two advantages of using OCL in system modelling compared to describing constraints only in natural language.

Show full model answer — Q1e
Model Answer — Q1e (4 marks)

Advantage 1 — Precision and unambiguity (2 marks)
Natural language is inherently imprecise. A requirement like "a student should not take too many courses" can be interpreted differently by different stakeholders — one developer may implement a limit of 3, another 5. OCL removes all ambiguity by expressing rules formally: self.enrolments->size() <= 4 has exactly one meaning. This prevents implementation errors arising from misunderstandings of the specification.

Advantage 2 — Formal analysis and automatic verification (2 marks)
OCL constraints can be checked automatically by model-checking tools and OCL validators. This means a tool can analyse whether a system design is consistent with its constraints before any code is written, catching design errors early. Natural language rules cannot be processed by such tools, making verification manual and error-prone.

Question 2 — Concurrency & Multithreading CRITICAL · 25 marks total

A food delivery platform processes thousands of orders simultaneously. Each incoming order is handled by a separate thread performing tasks such as verifying payment, assigning a delivery driver, and updating the restaurant order queue. During peak hours, many threads try to access shared resources (driver database, order queue) simultaneously.

Part a — Why synchronisation is needed (5 marks) · Time guide: 5 minutes

Show full model answer — Q2a
Model Answer — Q2a (5 marks)

Shared resources in multithreaded systems require synchronisation because multiple threads executing concurrently can access and modify shared data at the same time, leading to data corruption and inconsistent state. (2 marks)

In the food delivery system, the driver database is a shared resource. Suppose Thread A and Thread B both query the driver database to find a free driver. Thread A reads that Driver #7 is available. Before Thread A has finished assigning Driver #7, Thread B also reads the database and finds Driver #7 still marked as available. Both threads assign Driver #7 to different orders. The driver database now contains two conflicting assignments. Without a synchronisation mechanism (such as a lock), there is no guarantee that only one thread reads-then-writes the database at a time, making the outcome non-deterministic and incorrect. (2 marks)

Synchronisation mechanisms — such as mutex locks or semaphores — prevent this by ensuring only one thread can enter the critical section (the code that reads and writes the shared resource) at any moment. (1 mark)

Part b — Race condition (5 marks) · Time guide: 5 minutes

Show full model answer — Q2b
Model Answer — Q2b (5 marks)

A race condition is a defect in which the outcome of an operation depends on the relative timing or interleaving of multiple threads accessing shared data simultaneously. Because thread scheduling is managed by the operating system and is not predictable, the result of the computation can vary between runs — sometimes correct, sometimes corrupted. (2 marks)

In the food delivery order queue, a race condition could occur as follows: Order #42 is in the queue with a status of "pending". Thread A reads order #42 intending to assign a driver. Thread B also reads order #42 at almost the same moment (before Thread A has written back). Thread A updates the record to "driver assigned: Driver #3" and writes it back. Thread B, working from its stale copy, updates the record to "driver assigned: Driver #7" and writes it back, overwriting Thread A's update. The result is that Driver #3's assignment is silently lost — a corrupted record with no clear ownership. (2 marks)

The consequence is data corruption: incorrect driver assignment, lost updates, or duplicate assignments — all of which cause real-world failures (orders not delivered, drivers double-booked). (1 mark)

Part c — Two synchronisation mechanisms (6 marks) · Time guide: 6 minutes

Show full model answer — Q2c
Model Answer — Q2c (6 marks)

Mechanism 1 — Mutex Lock (3 marks)
A mutex (mutual exclusion lock) is a binary lock that only one thread can hold at any time. Before a thread enters the critical section (e.g., reading and writing the driver database), it must acquire the mutex. If another thread already holds it, the requesting thread is blocked and waits. Once the first thread completes its operation and releases the mutex, the waiting thread can proceed.

In the food delivery system, wrapping driver-assignment logic in a mutex ensures that only one thread checks and updates the driver's availability at a time. No second thread can read a stale "available" status while another thread is mid-update, eliminating the double-assignment race condition. Data consistency is maintained because the read–modify–write cycle is atomic from other threads' perspective.

Mechanism 2 — Semaphore (3 marks)
A semaphore maintains an integer counter. A thread calls wait (or P) to decrement the counter — if it reaches zero, the thread blocks. A thread calls signal (or V) to increment the counter, unblocking a waiting thread. A binary semaphore (initial value 1) behaves identically to a mutex. A counting semaphore (initial value N) allows up to N threads to access a resource concurrently.

For the food delivery order queue, a semaphore with an initial value of 1 could protect the queue so that only one thread updates it at a time. Alternatively, a counting semaphore could be used to limit how many threads can read the order queue concurrently, balancing access control with throughput. This maintains data consistency by ensuring updates to the queue are serialised.

Part d — Thread pool (5 marks) · Time guide: 5 minutes

Show full model answer — Q2d
Model Answer — Q2d (5 marks)

A thread pool is a collection of pre-created worker threads that remain alive and idle, waiting to execute tasks. When a new order arrives, it is placed in a task queue rather than spawning a new thread. An idle thread from the pool picks up the task, executes it, and then returns to the pool to await the next task. The number of threads in the pool is fixed (or bounded). (2 marks)

Advantage 1 — Reduces thread creation overhead (2 marks)
Creating a new OS thread for every incoming order is expensive: it involves memory allocation for the thread stack, OS scheduling registration, and initialisation. Under high traffic with thousands of simultaneous orders, this overhead significantly degrades performance. A thread pool eliminates this cost because threads are created once at startup and reused for many tasks.

Advantage 2 — Controls resource consumption (1 mark)
With a bounded thread pool, the system never spawns more threads than the pool size, preventing runaway memory consumption and excessive context switching even during peak traffic. This keeps the system stable and predictable under load.

Part e — Excessive threading harms performance (4 marks) · Time guide: 4 minutes

Show full model answer — Q2e
Model Answer — Q2e (4 marks)

Reason 1 — Context switching overhead (2 marks)
When the number of threads exceeds the number of available CPU cores, the operating system must rapidly switch between threads to give each one CPU time. Each switch requires saving the current thread's complete state (registers, program counter, stack pointer) and restoring the next thread's state. With excessive threads, the CPU spends a disproportionate amount of time on context switching rather than executing actual work — this is called "thrashing". Performance decreases even as thread count increases.

Reason 2 — Increased memory consumption (2 marks)
Each thread requires its own stack (typically 512 KB–1 MB on modern systems). Hundreds or thousands of threads consume gigabytes of memory. This can exhaust available RAM, causing the operating system to swap memory to disk (page thrashing), which is orders of magnitude slower. Additionally, large numbers of threads competing for shared cache lines increase cache contention, reducing the efficiency of each thread's execution.

Question 3 — Finite State Machines CRITICAL · 25 marks total

Part a — What is an FSM? (5 marks) · Time guide: 5 minutes

Show full model answer — Q3a
Model Answer — Q3a (5 marks)

A Finite State Machine (FSM) is a mathematical model of computation that can be in exactly one of a finite number of states at any given time. It transitions between states in response to inputs according to defined rules. FSMs are used to model sequential logic, protocols, compilers, and many other systems. (2 marks)

The five main elements of an FSM: (2 marks for listing all five)

  • Q — a finite set of states (e.g., {q0, q1, qDead})
  • Σ — a finite alphabet of input symbols (e.g., {a, b})
  • δ — a transition function: given a state and an input symbol, it returns the next state
  • q₀ — the start state (the state the machine begins in before any input is read)
  • F — a set of accepting (final) states

How they work together: The machine begins in q₀. It reads the input string one symbol at a time, left to right. After each symbol, the transition function δ moves the machine to a new state. When all symbols have been consumed, the machine checks whether the current state is in F. If yes, the string is accepted; if no, it is rejected. (1 mark)

Part b — DFA vs NFA (4 marks) · Time guide: 4 minutes

Show full model answer — Q3b
Model Answer — Q3b (4 marks)

DFA (Deterministic Finite Automaton): (2 marks)
In a DFA, for every state q and every input symbol a in Σ, there is exactly one transition. The machine is completely deterministic: given a state and a symbol, the next state is uniquely defined. There is no ambiguity or choice at any point. Every DFA always processes input in a single, deterministic computation path.

NFA (Non-deterministic Finite Automaton): (2 marks)
In an NFA, a state may have zero, one, or multiple transitions for a given input symbol. Additionally, an NFA can have ε-transitions (epsilon transitions) — moves to a new state without consuming any input symbol. When multiple transitions are possible, the NFA can be thought of as exploring all paths simultaneously. An NFA accepts an input string if there exists at least one path through the machine that ends in an accepting state. Every NFA can be converted to an equivalent DFA, but NFAs are often more compact to design.

Part c — Design a DFSM (6 marks) · Time guide: 6 minutes

Design a DFSM for L = { w ∈ {a, b}* | no two adjacent symbols are identical }. Include all states, start state, accepting states, and transition function.

Show full model answer — Q3c
Model Answer — Q3c (6 marks)

Insight: The only thing we need to remember is the last symbol read, so we can reject any string where the same symbol appears twice in a row. This requires 4 states.

States Q = { q0, qA, qB, qDead }
Alphabet Σ = { a, b }
Start state q0
Accepting states F = { q0, qA, qB }   (qDead is NOT accepting)

State descriptions:
  q0    — start state, no symbol read yet (empty string — accepted)
  qA    — last symbol read was 'a' (accepted so far)
  qB    — last symbol read was 'b' (accepted so far)
  qDead — two identical adjacent symbols seen (trap / reject state)

Transition function δ:

q0 (start)qAqB
qA (last was a)qDeadqB
qB (last was b)qAqDead
qDead (trap)qDeadqDead
 
State diagram (ASCII):

          a                b
   ┌────────────── q0 ──────────────┐
   ↓                                ↓
  qA ─── b ──→ qB          qB ─── a ──→ qA
   |                                |
   └── a ──→ qDead   qDead ←── b ──┘
             (qDead loops on both a and b — trap state)

Accepting states shown with double circle: q0, qA, qB
Rejecting (trap) state: qDead

Walkthrough example: Input "abab" → q0 →(a) qA →(b) qB →(a) qA →(b) qB. Final state qB ∈ F → ACCEPTED.
Input "aab" → q0 →(a) qA →(a) qDead →(b) qDead. Final state qDead ∉ F → REJECTED.

Part d — Explain the language (5 marks) · Time guide: 5 minutes

Show full model answer — Q3d
Model Answer — Q3d (5 marks)

The language L = { w ∈ {a, b}* | no two adjacent symbols are identical } contains all strings over the alphabet {a, b} where no two consecutive characters are the same. In other words, the string must alternate between the symbols 'a' and 'b' — no symbol may appear directly next to itself. The empty string is also in L (vacuously, there are no adjacent symbols to violate). (2 marks)

Two strings that belong to L: (1.5 marks)

  • "ab" — alternates a then b, no adjacent identical symbols.
  • "babab" — alternates b, a, b, a, b — every adjacent pair is different.

Two strings that do NOT belong to L: (1.5 marks)

  • "aa" — two adjacent 'a' symbols — violates the rule.
  • "abba" — the middle "bb" is two adjacent identical symbols — violates the rule.

Part e — What the DFSM must remember (5 marks) · Time guide: 5 minutes

Show full model answer — Q3e
Model Answer — Q3e (5 marks)

The DFSM only needs to remember the last symbol read. (2 marks)

This information is sufficient because the language's constraint is entirely local: it only compares each symbol with the symbol immediately before it. To check whether the current symbol violates the rule, we compare it to the previous symbol. If they are the same, we reject. If they differ, we continue. We never need to know any symbol further back in the string — only the most recent one matters. (2 marks)

This is why three states (qA, qB, and q0) plus a trap state are sufficient: qA encodes "last symbol was a", qB encodes "last symbol was b", and q0 encodes "no symbol seen yet". No additional history is required. If the language required remembering two or more symbols (e.g., "no three identical symbols in a row"), more states would be needed — but here, one symbol of memory suffices. (1 mark)

Object Constraint Language (OCL) CRITICAL

OCL is a text language that adds precise rules to UML models. It is not programming code — you cannot run it to do anything. Its job is to state what must be true, removing the ambiguity of natural language.

Why natural language fails: If a requirement says "a student should not take too many courses", one developer might implement a limit of 3 and another a limit of 5. OCL says exactly: self.courses->size() <= 4 — one meaning, no argument.

The three OCL constraint types — learn these precisely
  • Invariant (inv) — must be true at all times. Attached to a class with context ClassName inv Name:
  • Precondition (pre) — must be true before an operation runs. Attached to an operation.
  • Postcondition (post) — must be true after an operation completes. Uses @pre to refer to old values.

OCL Syntax Patterns

-- Invariant: always-true rule on a class
context Student
inv RuleName:
    self.courses->size() <= 4

-- Precondition: what must hold before an operation
context System::enrollCourse(s: Student, c: Course)
pre PreName:
    s.courses->size() < 4   -- strictly < 4, not ≤ 4 !

-- Postcondition: what must hold after the operation
post PostName:
    s.courses = s.courses@pre->including(c)
    -- @pre means "value before operation ran"

Structural vs Behavioural Constraints — the Q1d distinction

Structural

Constraints on the model's state. Written as invariants. Describe valid snapshots of the system at any moment.

Example: "A student always has ≤ 4 courses." This is a fact about the data — it must hold before and after every operation.

Behavioural

Constraints on operations. Written as pre/postconditions. Describe what the system must do (or what must be true) around an operation.

Example: "Before enrolling, the student must have no outstanding fees." This constrains the operation, not the standing data model.

Marks lost here

  • Writing pre/postconditions without a context line — loses marks for "proper syntax".
  • Forgetting postconditions entirely — the question always asks for both.
  • Using @pre in a precondition — it only makes sense in a postcondition.
  • Writing invariants as English sentences instead of OCL-style notation.

Concurrency & Multithreading CRITICAL

Concurrency is when a system handles multiple tasks during the same time period. A multithreaded program achieves concurrency by running several threads — each a separate line of execution — inside the same process. The problem: threads share the same memory, so they can interfere with each other.

Real-world analogy: Several chefs working in the same kitchen. They share the same fridge (shared resource). Without coordination rules (synchronisation), two chefs might both grab the last egg at the same time — one ends up with nothing and doesn't know it.

Core vocabulary — all of these appear in Q2
  • Thread — a unit of execution. Multiple threads share process memory.
  • Shared resource — data accessible by multiple threads (driver database, order queue).
  • Critical section — code that accesses a shared resource and must be atomic.
  • Race condition — outcome depends on unpredictable thread timing. Causes data corruption.
  • Mutex — a lock: only one thread holds it at a time. Others wait.
  • Semaphore — a counter. Limits how many threads access a resource simultaneously.
  • Monitor — a higher-level construct bundling a mutex + condition variables.
  • Thread pool — a fixed set of pre-created threads reused for many tasks.
  • Context switch — OS saves one thread's state and restores another's. Expensive.
  • Deadlock — threads waiting in a cycle — each holds what the other needs. All are permanently blocked.

How a race condition happens — step by step

  1. Thread A reads shared variable X (value: 5).
  2. Thread B reads shared variable X (also reads 5 — before A has finished).
  3. Thread A adds 1 and writes 6 back to X.
  4. Thread B adds 1 and writes 6 back to X — overwriting A's update.
  5. Result: X = 6, but should be 7. One update is lost silently.

Finite State Machines (FSMs, DFAs, NFAs) CRITICAL

A Finite State Machine is a model of computation. Think of it as a machine with a set of labelled rooms (states). It starts in one room, reads symbols from an input string one at a time, and moves between rooms according to rules. At the end, if it is in a "good" room (accepting state), the string is accepted; otherwise rejected.

Real-world analogy: A turnstile. States: Locked, Unlocked. Input: coin (→ unlock), push (→ pass through and lock again). It has a finite number of states and exact rules for each input.

The 5 elements of an FSM — memorise these exactly
  • Q — finite set of states
  • Σ (Sigma) — the alphabet (set of valid input symbols)
  • δ (delta) — transition function: δ(state, symbol) = next state
  • q₀ — start state (one only)
  • F — set of accepting (final) states (can be more than one)

DFA vs NFA

DFA — Deterministic

  • Every state has exactly one transition per symbol.
  • No choice — the next state is always uniquely determined.
  • No ε-transitions (free moves without reading a symbol).
  • Easier to implement as code; slightly harder to design for complex patterns.

NFA — Non-deterministic

  • A state may have zero, one, or many transitions for a symbol.
  • Can have ε-transitions (move without consuming input).
  • Accepts if any path through it ends in an accepting state.
  • Easier to design for patterns; can always be converted to an equivalent DFA.

Designing a DFSM — the Q3c approach

The language L = { w ∈ {a,b}* | no two adjacent symbols are identical } requires remembering only the last symbol read. This gives us 4 states:

States: q0 (start, no symbol yet), qA (last was 'a'), qB (last was 'b'), qDead (trap)
Accepting: q0, qA, qB   |   Rejecting: qDead

Transition table:
┌─────────┬────────┬────────┐
│  State  │   a    │   b    │
├─────────┼────────┼────────┤
│  q0     │  qA    │  qB    │
│  qA     │ qDead  │  qB    │  ← reading 'a' after 'a' → trap
│  qB     │  qA    │ qDead  │  ← reading 'b' after 'b' → trap
│ qDead   │ qDead  │ qDead  │  ← once dead, always dead
└─────────┴────────┴────────┘

Trace "abab": q0→(a)→qA→(b)→qB→(a)→qA→(b)→qB ∈ F → ACCEPT
Trace "aab":  q0→(a)→qA→(a)→qDead→(b)→qDead ∉ F → REJECT

Memory reasoning (Q3e)

Why remembering only the last symbol is sufficient The language's constraint is purely local — it only compares consecutive pairs. To decide if the current symbol violates the rule, we only need to know the previous symbol. We never need to know what happened two symbols ago. Therefore, one symbol of memory = three non-dead states is sufficient. If the language were "no three consecutive identical symbols", we would need to remember the last two symbols (more states).

Context-Free Grammars & Parsing CRITICAL

A context-free grammar (CFG) is a set of rewriting rules that define a language. You start with a special symbol (the start symbol) and repeatedly apply rules to replace non-terminal symbols with other symbols until only terminal symbols (the actual string characters) remain. This process is called derivation.

Analogy: A recipe. "Meal → Starter Main Dessert". "Main → Pasta | Steak". You keep expanding until you have actual food names — those are the terminals.

Grammar vocabulary
  • Terminal — a symbol that appears in final strings (e.g., id, +, 0, 1)
  • Non-terminal — a placeholder that gets replaced (e.g., E, T, F, S)
  • Production rule — a replacement rule written A → α (replace A with α)
  • Start symbol — the non-terminal you begin from (usually S or E)
  • Derivation — the sequence of rule applications producing a string
  • Leftmost derivation — always expand the leftmost non-terminal first
  • Parse tree — a tree showing the derivation; root = start symbol, leaves = terminals

Leftmost derivation — worked example (Q4a)

-- Grammar: E → E+T | T   T → T*F | F   F → (E) | id
-- Derive: id + id * id (leftmost = always expand leftmost non-terminal)

E
→ E + T         Apply E → E+T  (leftmost E expanded)
→ T + T         Apply E → T    (leftmost E expanded)
→ F + T         Apply T → F
→ id + T        Apply F → id
→ id + T * F    Apply T → T*F
→ id + F * F    Apply T → F
→ id + id * F   Apply F → id
→ id + id * id  Apply F → id  ✓ done

How grammars encode precedence

The non-terminal hierarchy encodes which operator binds tighter. In the grammar above, * lives inside T (lower level), while + lives inside E (higher level). Because T must be fully resolved before it can be an operand of E's + rule, multiplication always happens before addition — exactly like BODMAS.

Parse tree for "0011" from S → 0S1 | 01 (Q4b)

        S                   Derivation:
       /|\                  S → 0 S 1       (apply S → 0S1)
      0  S  1                    → 0 0 1 1  (apply inner S → 01)
        / \
       0   1              Leaves: 0, 0, 1, 1 = "0011" ✓
Parse tree drawing tips Draw the root (S) at top. Each production is one level of the tree. Terminals are the leaves (bottom). The examiner checks that the tree structure correctly reflects the grammar rules — labels matter.

Regular expressions vs CFGs — the limitation (Q4e)

What regular expressions cannot do Regular expressions correspond to finite automata (no memory). They cannot count or match nested structures. The canonical example: balanced parentheses (((()))) — a regex cannot verify the brackets match because it cannot count. This requires a CFG (and a pushdown automaton with a stack). Most programming language constructs — nested blocks, matched tags, recursive calls — fall into this category.
Common mistakes in Q4
  • In leftmost derivation, expanding a right-side non-terminal before the left one — marks deducted for wrong step order.
  • Drawing a parse tree as a list rather than an actual tree diagram with branching.
  • For "strings of length 4 from S → 0S1 | 01": writing 0101 or other strings — only 0011 is correct.
  • For regex limitations, saying "regex can't match everything" without a concrete example — always use balanced brackets.

Lexical analysis — the three tasks (Q4c)

  1. Scan — read the source code character by character, left to right.
  2. Tokenise — group characters into tokens: keywords, identifiers, literals, operators, delimiters. Whitespace and comments are discarded.
  3. Output token stream — pass the classified tokens to the parser for syntax analysis.

Symbol table (Q4d)

What a symbol table stores and why it matters

A symbol table is a data structure (often a hash table) that maps every identifier to its properties:

  • Name — the identifier string (e.g., "totalCost")
  • Type — e.g., int, float, String
  • Scope — where the identifier is valid (local to a function, global, etc.)
  • Address/offset — memory location for code generation

Why important: Semantic analysis uses it for type checking ("can you add an int and a String?"). Scope resolution uses it to find which declaration applies in nested blocks. Code generation uses it to emit the correct memory instructions.

Bottom-up parser (Q4f)

A bottom-up (LR) parser starts from the input tokens and works upward — applying grammar rules in reverse (reductions) until the entire input reduces to the start symbol. Key advantage: handles LR grammars including left-recursive grammars, which covers almost all real programming languages.

Common mistakes in Q4 compiler parts
  • Confusing lexical analysis with parsing — lexical analysis produces tokens; parsing checks their syntactic structure.
  • Saying the symbol table "stores the code" — it stores identifiers and their properties, not executable code.
  • For tokenisation, describing only keywords and ignoring identifiers, literals, operators, delimiters — aim for at least 3 categories.

Software Quality & Testing CRITICAL

Software quality is about how well software meets its requirements and how reliably it does so. For safety-critical systems (traffic lights, medical devices, aircraft), quality is not optional — failures cause accidents. Q5 tests your ability to apply quality concepts to a concrete scenario.

Functional vs Non-functional testing — the Q5c distinction
  • Functional testing — checks what the system does. Inputs → expected outputs. "Does the light change when sensor detects high traffic?"
  • Non-functional testing — checks how well the system does it. Performance, reliability, security. "Does the light change within 200ms under peak load?"

Verification vs Validation

Verification

"Are we building the product right?" — checking the system against its specification. Does the code do what the design says?

Methods: code reviews, static analysis, model checking, formal proofs.

Validation

"Are we building the right product?" — checking the system meets actual user/stakeholder needs. Does the product solve the real problem?

Methods: user acceptance testing, simulation, beta testing.

Three V&V techniques for the traffic system (Q5d)

  1. Simulation/Scenario Testing — construct realistic urban traffic scenarios including edge cases (sensor failure, peak hour surge, pedestrian crossing). Validate that the system responds correctly and safely without real-world risk.
  2. Formal Verification / Model Checking — mathematically verify safety properties (e.g., opposing lights never both green). Exhaustively checks all possible states — proves absence of certain bugs, not just absence in tested scenarios.
  3. Code Reviews / Safety Audits — independent engineers review code and design for hazards, untested paths, and compliance with safety standards (e.g., IEC 61508). Catches issues automation misses.

Fault tolerance strategies (Q5b)

Two design strategies for fault tolerance
  • Redundancy — duplicate critical components (sensors, processors). If one fails, others continue. No single point of failure.
  • Graceful degradation — when a component fails, the system switches to a safe fallback mode rather than crashing. E.g., fixed-cycle timing when sensors fail. The system continues operating at reduced capability.

Unit testing in high-integrity software (Q5f)

Unit testing verifies individual components in isolation. In high-integrity software, it allows faults to be detected and isolated at the earliest possible point — before they propagate through the system and become catastrophic. Early detection is cheaper and safer than finding faults after integration.

Limitations of a single metric (Q5e)

Complex quality attributes (maintainability, reliability) have many dimensions. A single metric captures only one. Example: using LOC to measure maintainability ignores readability, modularity, and test coverage. Using MTBF to measure reliability ignores failure severity and recovery time. Multiple metrics + qualitative judgement are needed.

Common mistakes in Q5
  • Answering Q5 as pure theory without linking to the traffic light scenario — the mark scheme says "illustrate with the system". Always say "in the traffic control system, this means…"
  • Confusing verification with validation — verification = against spec, validation = against user need.
  • For fault tolerance, saying "add more testing" — testing is not a design strategy. Testing checks; redundancy and graceful degradation are design strategies.
  • For the single metric question, describing only one example instead of explaining the general principle plus an example.
5 Quick Quiz Questions
  1. What is the difference between functional and non-functional testing? Give one example of each for the traffic system.
  2. What is the difference between verification and validation? One sentence each.
  3. Name three V&V techniques for a safety-critical system.
  4. What is graceful degradation? Give an example from the traffic system.
  5. Why is a single metric insufficient to evaluate reliability?
Show answers
  1. Functional: tests what the system does — e.g., verifying the light changes to red when sensor detects no vehicles. Non-functional: tests how well — e.g., testing response time is under 200ms under peak load.
  2. Verification: checking the system matches its specification. Validation: checking the system meets actual user needs.
  3. Simulation/scenario testing; formal verification/model checking; code reviews/safety audits.
  4. Graceful degradation means continuing to operate in a reduced-capability safe mode when a component fails. Example: if a sensor fails, the traffic light switches to fixed-cycle timing instead of shutting down.
  5. Reliability is multi-dimensional — it includes failure frequency, severity, and recovery time. A single metric like MTBF measures frequency but ignores severity. A system failing rarely but catastrophically has a high MTBF but low real-world reliability.

Software Metrics & GQM MEDIUM

Software metrics are quantitative measures of software properties — e.g., lines of code, defect density, test coverage. The GQM (Goal–Question–Metric) framework provides a principled way to select metrics that actually answer the questions you care about.

GQM Framework
  1. Goal — define what you want to achieve or understand. E.g., "Improve the reliability of the traffic control software."
  2. Question — what do you need to know to assess whether the goal is met? E.g., "How often does the system fail?" "How quickly does it recover?"
  3. Metric — what number answers each question? E.g., Mean Time Between Failures (MTBF), Mean Time To Recovery (MTTR).

CI/CD stands for Continuous Integration / Continuous Delivery (or Deployment). It is a practice where code changes are automatically built, tested, and deployed as soon as developers commit them — rather than in infrequent large releases.

Before you write anything

  2. Mark which 3 you will answer. Choose based on which mark schemes you can satisfy, not just topic familiarity.
  3. For each chosen question, quickly bullet-list your main points before writing prose — this prevents mid-answer confusion.
